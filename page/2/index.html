PKU ACM 1013 Counterfeit Dollar

发表于 2016-03-23   |  

题目大意

有标记着A到K的12个硬币，其中有一个假币。秤重三次，只能看出两边是否平衡。求哪个编号对应的硬币是假币，且求其轻重。

梳理

其实是个简化版的13球问题（相对其它12个球重量不同，3次求出，找称重的方法）。
由于这次称重的办法都由题目提供，所以只要很简单地输出判断结果就好啦。
给出一个新类型Coin。
Coin
char letter;//字母
enumType type;// true mayup maydown none
void setType(_type){
如果我是up，目标是down，那么就设成true，反之_type
}

坑

  1. 输入的获取。本来想按字符判断输入的，然而试了半天Java的System.in.read()，会把新行和回车区别对待造成识别与预期不同。最后用Scanner.next()+String.substring()拆一段字符= =
  2. 判断的错误。通过setType方法确实已经完成了大部分轻重判断，但忽略了一个点——两个硬币对比出现轻重不同时，其他硬币就全都是真。加入这步判断，人脑的尝试和计算机处理算完全对上了。
  3. 最后只要输出列表中第一个不是True的对象就可以。

PKU ACM 1012 Joseph

发表于 2016-03-18   |  

题目大意

  • 有n个人，编号1,2,……,n。
  • 报个数m，所有人围成一圈1到m抱树，抱到m的去死。其他继续，直到最右一个放生。
  • 现在有k*2个人，前一半是好人，后面是坏人。要求一个m，让坏人先死光。（0 < k < 14）

个人想法，既然只有13种情况，枚举完留个表给PKU好了。

忙了两天，回来静下来总结。这题有一些平时没注意的地方。

  1. 最基本的计数方式，数人头，其实有两种逻辑——从第一个人开始数，杀了一个人后从下一个人开始数。这个表述方式在数组里显得太麻烦，我就改用链表了。
  2. 剪枝prune。一开始只减了一个，就是砍到好人就return，没什么好说的。后来想想，如果这样需要剪掉的话，直接从枚举过程中去掉就好了。
    根据讨论区的总结，m=(k+1)i || m=(k+1)i+1。并没有推出为什么。从结果上来说，进行这一步剪枝后，仍然超时，他的意思就是让我们直接返回数组咯，反正只有13种情况。

    总结

    回忆了一下链表的知识，大概还能捋清楚头尾相连之类的。

美术：廖毅（《汐》作者）、seedsman

游戏简介

寓意

主人公所遇到的困境会是目前的他没有办法解决的，但只要你一直走不回头，世界会给你让出一条路。

特色

在烂大街的横版跳台基础上，加上了不太烂大街的4方向重力转换，以及独创的第5重力方向（垂直于屏幕，就是推箱子）与世界线（第5重力于4向重力切换）。
游戏中同时包含操作难度和解密难度。如图1中，推箱子重力下地面的尖刺无法通过，需要推动箱子来盖住尖刺。又如图2中，跳台中的尖刺有很微妙的碰撞判断，需要精密的操作来通过。

Theme"We don't see things as they are, we see them as we are."

想只靠游戏的玩法方式来体现主题很有难度。那是个人第一次参加Gamejam，发觉大家对主题的表达都不是很好，大多都是靠演示的人来口头总结，并没有通过游戏体验到。

GlobalGamejam 2016 救赎圣经（Redemption Bible）

发表于 2016-03-15   |  

小伙伴

程序：sommily、刘城
美术：shingle、penny、seedsman
策划：穆风
音效：老K

游戏简介

“你”为了洗清身上的罪孽，在地狱接受救赎的刑罚。 玩家必须真心诚意地接受惩罚，感到疼痛时叫喊，被烫伤时甩手，表现得足够痛苦后地狱的使者才会放你过关。 【特色：声控、体感。游戏需要配合“叫喊”和“甩手”进行操作】

演示视频（密码:GGJ2016)


这是一款定位成惩罚活动的游戏。聚会的时候如果想看别人出丑就可以嘿嘿嘿~~


如图，演示视屏中没有来得及展示第二关。第二关开始会有火焰攻击，手上着火后需要上下摆动手臂，触发体感反馈来熄灭火焰。

设计思路

充分理解了程序和美术的热情后，我找到了同时满足彼此特长的方案，决定制作一款让现场各位都能开心的游戏。
主题Ritual
讨论之初美术提到了刑罚、诡异、七宗罪，程序说自己可以提供声控和体感。于是大致的方向就有了，我们可以做一款体感游戏。
然而讨论过程中，美术妹子过于沉溺于体感操作的新鲜感，提出了一些只是依靠体感来操作的传统游戏模式。这个思路并没有什么不妥，否认这个思路并非因为不够创新，而是无法掌控体验感。

如果用体感替代传统游戏的操作，那操作仍然是重点。要用体感表现操作没有任何问题，我完全相信程序的实力，然而操作的内容却是一个问题。有经验的人都知道，最近打斗画面绚丽的游戏越来越多了，尽管看起来打斗漂亮，玩起来却很不痛快。这是因为缺乏体验曲线，全程都是很高亢的华丽斩杀，打小怪是这么几刀还好，打BOSS时还是这么几招就没有价值。实际游戏里肯定只有很少几种情况才能打出华丽的操作，不但要做出美好的高潮，还要做细腻的前期铺垫。如此细致就无法去关注体感和美术内容。

我个人则更倾向于用简单的游戏方式，来突出体感的特长——代入感。制作方式就是简化操作，并铺开很大的量组做反馈内容，看起来就像太鼓达人。太鼓达人是一个十分优秀的例子，虽然操作只有“咚”“呛”两种，但反馈非常丰富。
太鼓达人中每按一下按钮：

  • ·音符轨道会变色
  • ·音符轨道的重点会有三种命中程度提示
  • ·命中时会有积分特效飞向分数文字
  • ·命中时会有与歌曲主题有关的人偶飞向终点
  • ·未命中时会有灰色的灵魂飞向终点
  • ·屏幕下方的太鼓图像会变色
  • ·50连击以上，轨道上飞来的音符会有动画
  • ·不同歌曲的不同连击数达成后会在屏幕下方显示不同的角色动画

以这种形式来制作，可以减轻核心内容的压力，并且周边铺得越多感受越好。
于是我提出用做一个被鞭打时要叫喊的模式。
于美术：契合刑法和七宗罪，让她们能自由发挥魔法少女小圆的画风。
于程序：充分展示声控和体感操作的技术。
于我：有效表达了很黄很暴力的主题，而且从演示效果来看，喊“啊”的时候绝对很羞耻。

总结

体感操作时，操作的准确性对体验影响非常大。既玩家是游戏的一部分，如果玩家没有做到响应的程度，效果也会大大折扣。
这就是被鞭打时，喊“啊~”能给大家带来笑点的原因。后期改喊“爽”就是分出戏。
这点实际上看来还是和生活密切相关。就像一群小朋友玩老鹰抓小鸡，掺入一个不害怕被老鹰抓住的小鸡，对于那个孩子游戏就完全没有意思。
不论工作还是生活，都需要很高的切合度。

运行环境

游戏部分是C++写的，语音控制部分是python写的。

注意事项：

如果需要语音控制，需要安装Python的开发环境，Mac自带python2.7 所以只需要下载 pyaduio即可（pip install pyaudio）
如果不需要控制，只需要按 键盘 “J”，作为触发发声动作即可。

PKU ACM 1011 Sticks

发表于 2016-03-12   |  

逗比乔治撸了几个长度一样的管。不小心撸断了，碎成好几段。现在要从碎掉的几段里，推测原本管子长度最短可能是多少。
简单说就说很富杂的东西。
可以穷举搞定，有几个地方能优化。

  1. 最长的碎片，说明原始长度不可能比他短。

  2. 碎片总和，原始长度不会比他长。

    **坑点**
. 验证数组中的组合能满足猜测的长度。需要尝试多种长度，多种方式的组合
. 递归结果的返回，有很多种解，都要尝试。会出现用掉别人的parts导致加不出的情况。
2016年3月12日12点36分43秒
看了会片回来，觉得有三层递归。
最底层，寻找满足求和方式的组合
每种不同的求和方式，都要开一条线检测后续能否继续剪完
最上，尝试各种长度

经过一番挣扎，我的做法还是在空间和时间复杂度上都极其不优雅。虽然题目只报了超时，但估计内存也会超。
查看别人的代码后总结如下优化点

寻找求和组合的方式

我很傻的使用了独立数组去存储每种求和方式，然后把数组返回过来，再遍历一次这个数组从总sticks里去掉重复项。
参考的代码只用了一个数组，标记true/false，true表示用过，false没有。进行下一层递归前把用掉的数字设true，反正递归结束发现这个做法不行时会设会false。

完成一个长度的验证

原本我是检查是否所用sticks都用过，然而如果没次都检测所有的格子是否是true，那又是一个n^2以上。
参考的算法里，通过设定单个长度，和总长相除获得了全根数。只要每次找到一种合适的加法，把已找到的根数+1就可以判断是否完成人物。

已知组合的搜索跳过

在求和检测中，一次for循环里，如果当前数字无解，那么相同数字在当前遍历中也是无解的。
本来相对这个优化举一反三，让这次遍历中，对相同的数字全部跳过。

建立跳过表

与标记是否使用过的表格一样，当检查到无法使用的数字时：从当前位置开始向后，把值设为true。同一次遍历中，不再检测相同的数字。
但下一次递归中就要重置，因为状态变化了，所以一张表只在一次遍历中有用。那么就会造成更大的空间复杂度。

其实这个问题已经在排序中解决过了。

最后使用的方法

在查阅资料后，注意到几点
· 数据用降序排列效率更高 从头几支的搜索来分析，先搜索大的数可以减少深度。具体并不清楚。
· 基于降序排列，原本升序数组可以在==判断后剪枝的内容不可行。>的情况同理 原来下一位会比现在大，永远不满足判断条件。
· 如果搜索中没有找到结果，也要把总长输出来（只有一根的情况）
· _sum，就是当<情况时传递的参数。如果两个判断之后 _sum==0，就跳过后续的遍历。为什么呢？它象征的是空手开始找线索。他从两种意义上来说都是必然有结果的。

1. 一根棒子必然属于正确解的一部分。（这样理解容易产生误差，并不排除因为组合问题导致一根棍子找不到配对的情况）
2. 因为列表中的每一项都是解的一部分，从哪一只都应该能找到正确解。所以正确的来说只需要一个起点，并走到底。

    综上，在同一次遍历中，只需要随便一跟都是对的。但如果手上已经拿这一根，但找不出答案，那可能是第二个根拿错了。

坑自己

之前在一份C语言的参考代码中看用while跳过重复数据的写法，在学习过程中写错，对每次跳过都写了一次赋值，消耗的量比较大。
在最后的优化中，考虑到每次遍历记录skip下标的赋值操作是不是也会很耗，于是写了个while式的。其实没什么卵用，用64个的那组数据测试比原来写法平均长200ms。
这里的原因是仍然是数量级问题，因为之前写错的时候是在一次搜索里赋值好几次。而skip的方式同样是一个判断，一个赋值，甚至不用考虑 下标越界。
