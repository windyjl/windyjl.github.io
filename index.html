动态规划 背包问题

发表于 2016-03-26   |  

主要参考了 Hawstein的动态规划之背包问题（一）

动态规划（DynamicProgramming）

所谓动态规划，旨在通过上一个问题的解来求下一个问题的答案。感觉就像打死了一直BOSS后，获得了这个BOSS身上的提示，然后用来解开谜题。有时候需要先打死很多个BOSS，等解密的时候再去翻尸体。这一点在之后的代码中会体现出来。

正式搭上动态规划这辆车之前，需要认识几个路标：状态，状态转移方程。

状态

一般存储动态规划解的数组命名为dp，每一个dp[i]、d[i][j]、d[i][j][k]都是一个状态。

借用参考资料的例子，有n个宝石，第i课宝石体积为v[i]，价值为w[i]。使用前n种宝石，体积限制为C时，能装下的最高价值的宝石组合的最高价值为dp[n][C]。

dp[1][10]的意思就是我只用第1颗宝石，在体积限制为10的情况下，能赚的最多的价值。
为什么要用前n种宝石这种方式呢？这跟我们状态转移方程的使用方式有关。

状态转移方程

状态转移方程的元素有二

  1. 前置状态—— dp[i-1][j-v[i]——未考虑新宝石时，剩余体积能满足新宝石加入的解
  2. 到当前状态的差值——w[i]——加入新宝石的变化

套入范例，如果我们要看加入新宝石后的最高价值，只要结合1、2两点的和，再与未考虑新宝石的最高价值对比。求出dp[i][j]的值。
原文充份讲解了状态和转移方程的含义和使用，但转移方程的推导不甚详细，特此说明。并提醒自己和其他读者注意方程的推理过程，注意逻辑严密性。参考文章着重说明了动态规划的整体，但没有针对文中例子进行完整的推导。实际运用时并非只需要关注方程的存在，还要找到合适的方程。
按照这个思路，我们要找到在n个宝石中，选取总体积为C的宝石，求最大值。就要找到比n或者C早一些的状态。（当然这里明确指出n和C有些作弊，已经提出了2个变量。具体问题思考时，也要分析哪些值用来表示状态）
案例中我们的做法就是求出第i个宝石加入前的最优解，然后找出满足第i颗宝石放入条件的前置状态。放入宝石。对比放入宝石后的值与同样体积时i-1的值，求出新的解。

完整推导

题目

PKU ACM 1014 Diving
同样是宝石，总共6种，每种若干个（小于两万）。求能不能平均分。
也就求能否求出和是sum/2。

因为是求sum/2有没有解，那么可以用dp[i]>0表示有解，i表示当前的和。
每个宝石的价值为w[j]。每个宝石的数量为m[j]。
dp[i+w[j]] 需要dp[i]有解。且m[j]使用的量不超标。
从dp[0]开始，把m[j]保存上去，dp[w[j]]=dp[0]-1;
如此状态转移方程的元素应该就满足了。

  1. 前置状态——dp[i]保存剩余宝石数量
  2. 到当前状态的差值——1

修正

跑了一圈程序，发现刚刚的推理在实际操作中有问题。以上面的方式来说，每次找到合适的前置状态，剩余的石头都-1.这样是不对的，实际情况中能在所有有解的dp后面重新开一条递减。

经过一个蹲坑的时间的思考，重新推理了一下。按照遍历过程，当dp[i]中的i不断增大，遇到dp[i]>0时，必然是之前的宝石留下起点。故判断dp[i]>0时，将其值赋值为宝石数量，进行新一轮递增。

不用担心

虽然我思考过程中不断担心会不会在同一次遍历中，当前宝石的剩余数值干扰计算，但事实上因为顺序为题，这种情况并不会发生。他们会被早一步设成新的起点。

经过几轮调试，总算改对了，用dp解出了这个问题。
其中一个坑居然跟我的初始化有关，因为Java自动把数组初始化为0。那么消耗掉dp[i-w[j]]>0上的宝石得到的dp[i-w[j]]-1==0时会被判断为无效。而偷看的别人的代码是用-1初始化的。真是意外XP
于是偷懒地把所有的数值和判断都加了1。

意外收获

之前思考这题的时候，总觉得当一种宝石超过一定数量后，后续的数量除了奇偶性是没有意义的，但没有充分的理由。用动态规划的状态来看的话就十分明显，例如1就很明显，他必定能在6次之内与其它数重合，开启新起点。借此优化输入数据可以更快求得解。

剪枝

从前一次遍历的最后一个结开始，项目退最小公倍数-宝石价值，作为下一次遍历的起点。因为存在公倍数概念，每一个最小公倍数周期都会遇上能作为起点的解，这个位置满足所有分歧类型的需要。
时间复杂度几乎降为sum/2。

实验结果

在pku上提交带剪枝的代码，计算时间反而加长了，剪得越精细越长ヾ(｡｀Д´｡)

—|—|—

|很粗暴的统一减120时 | 329ms |
|仔细地按数组剪掉必要长度时（代码部分）| 344m |

PKU ACM 1014 Dividing

发表于 2016-03-23   |  

题目大意

输入n1 n2 …… n6，每个位置代表对应分值有多少数量。要平均分两份。

梳理

  1. 首先总和一定要是偶数
  2. 找出一种方法，其和是总值的一半。这个跟之前做的stick很像。（如果是这样，那么主要工作量在剪枝）

中途做了一半重新梳理

虽然用深度搜索可以完成工作，但是面对稍微大量的数据就会十分缓慢。所以决定拆了重写。
首先确定如果每一位上都是偶数，那么肯定是能够平均分的。
如果某一位上是奇数，则拆借其他位置上的数来补充。
补充时，以对应数字的1倍到最大公倍数60为跨度。比如6分的石头是奇数个，那么用5补充时，发现配不平，则采用12,18,24,30。30的时候发现平了等等。
暂时称为借位搜索。

最后梳理

前一版提出的算法并不成熟，当然一个下午想闭门造出新的算法也很不实际。最后提交版本用的是优化输入数据的办法，继续深度搜索。
因为各个数字超过一定数量后，都可以用他们本身的数量去填平其他数字不足的地方。比如我有100个1，那其他数字再怎么折腾也能配得平了。
那具体多少个呢？严谨地说没算出来。有人建议是最小公倍数/value,也就是代码中的值。

然而并没有结束，我最后需要研究下**动态规划**求另一种解法。

PKU ACM 1013 Counterfeit Dollar

题目大意

有标记着A到K的12个硬币，其中有一个假币。秤重三次，只能看出两边是否平衡。求哪个编号对应的硬币是假币，且求其轻重。

梳理

其实是个简化版的13球问题（相对其它12个球重量不同，3次求出，找称重的方法）。
由于这次称重的办法都由题目提供，所以只要很简单地输出判断结果就好啦。
给出一个新类型Coin。
Coin
char letter;//字母
enumType type;// true mayup maydown none
void setType(_type){
如果我是up，目标是down，那么就设成true，反之_type
}

坑

  1. 输入的获取。本来想按字符判断输入的，然而试了半天Java的System.in.read()，会把新行和回车区别对待造成识别与预期不同。最后用Scanner.next()+String.substring()拆一段字符= =
  2. 判断的错误。通过setType方法确实已经完成了大部分轻重判断，但忽略了一个点——两个硬币对比出现轻重不同时，其他硬币就全都是真。加入这步判断，人脑的尝试和计算机处理算完全对上了。
  3. 最后只要输出列表中第一个不是True的对象就可以。

PKU ACM 1012 Joseph

发表于 2016-03-18   |  

题目大意

  • 有n个人，编号1,2,……,n。
  • 报个数m，所有人围成一圈1到m抱树，抱到m的去死。其他继续，直到最右一个放生。
  • 现在有k*2个人，前一半是好人，后面是坏人。要求一个m，让坏人先死光。（0 < k < 14）

个人想法，既然只有13种情况，枚举完留个表给PKU好了。

忙了两天，回来静下来总结。这题有一些平时没注意的地方。

  1. 最基本的计数方式，数人头，其实有两种逻辑——从第一个人开始数，杀了一个人后从下一个人开始数。这个表述方式在数组里显得太麻烦，我就改用链表了。
  2. 剪枝prune。一开始只减了一个，就是砍到好人就return，没什么好说的。后来想想，如果这样需要剪掉的话，直接从枚举过程中去掉就好了。
    根据讨论区的总结，m=(k+1)i || m=(k+1)i+1。并没有推出为什么。从结果上来说，进行这一步剪枝后，仍然超时，他的意思就是让我们直接返回数组咯，反正只有13种情况。

    总结

    回忆了一下链表的知识，大概还能捋清楚头尾相连之类的。

小伙伴

美术：廖毅（《汐》作者）、seedsman
关卡策划：穆风（本人）

游戏简介

寓意

主人公所遇到的困境会是目前的他没有办法解决的，但只要你一直走不回头，世界会给你让出一条路。

特色

在烂大街的横版跳台基础上，加上了不太烂大街的4方向重力转换，以及独创的第5重力方向（垂直于屏幕，就是推箱子）与世界线（第5重力于4向重力切换）。
游戏中同时包含操作难度和解密难度。如图1中，推箱子重力下地面的尖刺无法通过，需要推动箱子来盖住尖刺。又如图2中，跳台中的尖刺有很微妙的碰撞判断，需要精密的操作来通过。

制作总结

Theme"We don't see things as they are, we see them as we are."

想只靠游戏的玩法方式来体现主题很有难度。那是个人第一次参加Gamejam，发觉大家对主题的表达都不是很好，大多都是靠演示的人来口头总结，并没有通过游戏体验到。

真的要表达这个效果的话，恐怕使用文字描述比玩法展示要好得多。
